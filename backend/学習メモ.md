# TypeScript 学習メモ（Phase 0〜5）

## 1. `type` 定義では値の代入ができない

**Q:** `type` の中で `roles = ["ADMIN", "STAFF", "USER"]` と書いたらエラーになるのはなぜ？

**A:** TypeScript の `type` は**型の形（構造）を定義するもの**であり、具体的な値を代入することはできない。取りうる値を制限したい場合はユニオン型を使う。

```typescript
// NG: 値の代入はできない
type Users = {
    roles = ["ADMIN", "STAFF", "USER"];
}

// OK: 型の制約として表現する
type Users = {
    roles: ("ADMIN" | "STAFF" | "USER")[];
}
```

---

## 2. `string[]` とユニオン型の違い

**Q:** `roles: string[]` と `roles: ("ADMIN" | "STAFF" | "USER")[]` は何が違う？

**A:**

| | `string[]` | `("ADMIN" \| "STAFF" \| "USER")[]` |
|---|---|---|
| 任意の文字列 | 入れられる | 入れられない |
| タイプミス防止 | できない | コンパイル時にエラーになる |
| 厳密さ | 緩い | 厳密 |

ロールのように決まった値しか取らないものには、ユニオン型を使う方が安全。

---

## 3. 型の分類名

**Q:** ユニオン型に対して `string[]` のほうは何という？

**A:**

| 型 | 分類 | 例 |
|---|---|---|
| `string` | プリミティブ型 | 任意の文字列 |
| `string[]` | 配列型（プリミティブ型の配列） | 任意の文字列の配列 |
| `"ADMIN" \| "STAFF"` | ユニオン型（Union Type） | 特定の値のみ |
| `"ADMIN"` | リテラル型（Literal Type） | その値だけ |

広い型を**ワイド型（Wide Type）**、狭い型を**ナロー型（Narrow Type）**とも呼ぶ。型を狭めることを**型の絞り込み（Narrowing）**と言う。

## 4. `sqliteTable()` 関数とは

**Q:** `sqliteTable()` 関数とは何？

**A:** Drizzle ORM が提供する関数で、TypeScript のコードで DB テーブルを定義できる。`type` と違い、型情報と DB 定義の両方を担う。

| | `type` | `sqliteTable()` |
|---|---|---|
| TypeScript の型情報 | 自分で書く | 自動で生成される |
| DB テーブルの作成 | できない | できる |

基本構文：
```typescript
import { sqliteTable, text, integer } from "drizzle-orm/sqlite-core";

const products = sqliteTable("products", {
    id: text("id").primaryKey(),
    name: text("name").notNull(),
    price: integer("price").notNull(),
});
```

カラムの型関数：`text()` → 文字列、`integer()` → 整数、`real()` → 小数
カラムのオプション：`.primaryKey()`、`.notNull()`、`.default(値)`

## 5. `InferSelectModel` と `InferInsertModel`

**Q:** これらは何のためにある？

**A:** スキーマから TypeScript の型を自動で作ってくれるユーティリティ型。手書きの二重管理を防ぐ。

| | `InferSelectModel` | `InferInsertModel` |
|---|---|---|
| 用途 | DB から取得したデータの型 | DB に挿入するデータの型 |
| 特徴 | すべてのカラムが含まれる | デフォルト値のあるカラムは省略可能 |

SELECT 時には `id` は必ず含まれるが、INSERT 時には自動生成なら渡さなくても良い。だから2つに分かれている。

`import type` は「型だけをインポートする」構文。実行時の JS には含まれず、コンパイル時にだけ使われる。

## 7. マイグレーションとは

**Q:** マイグレーションって何？

**A:** データベースの構造変更を管理する仕組み。Git がコードの変更履歴を管理するように、マイグレーションは DB 構造の変更履歴を SQL ファイルとして残す。

全体の流れ：
```
① schema.ts（TypeScript のコード）
    ↓  drizzle-kit generate
② マイグレーションファイル（SQL 文）
    ↓  drizzle-kit migrate
③ DB にテーブルができる
    ↓  seed.ts
④ テーブルに初期データが入る
```

---

## 8. `drizzle-kit push` vs `generate` + `migrate`

**Q:** 3つのコマンドの違いは？

**A:**

| コマンド | やること | マイグレーションファイル |
|---|---|---|
| `drizzle-kit push` | スキーマの変更を直接 DB に反映 | 作らない |
| `drizzle-kit generate` | SQL ファイルを生成 | 作る |
| `drizzle-kit migrate` | 生成済み SQL を DB に適用 | 既にあるものを使う |

`push` はローカルで試行錯誤中に便利。`generate` → `migrate` はチーム開発や本番デプロイで使う。

**実行する場所：** `drizzle-kit` のコマンドは `package.json` と `drizzle.config.ts` がある階層（このプロジェクトでは `backend/`）で実行する。`drizzle-kit` は `drizzle.config.ts` を読みに行くため、その設定ファイルがあるディレクトリにいる必要がある。

## 9. `.env` で DB パスを指定する理由

**Q:** なぜ `.env` を使うのか？

**A:** DB のファイルパスは環境によって変わる可能性がある（開発・テスト・本番）。コードに直接書く（ハードコード）と環境が変わるたびにコードの書き換えが必要。`.env` に切り出せばコードを変えずに設定を切り替えられる。

## 10. `tsconfig.json` の役割

**Q:** `tsconfig.json` は何をしている？

**A:** TypeScript コンパイラ（`tsc`）への指示書。「TypeScript のコードをどうやって JavaScript に変換するか」のルール。

| 設定 | 意味 |
|---|---|
| `target` | 変換先の JavaScript バージョン |
| `module` | モジュールの仕組み |
| `strict` | 型チェックを厳しくする |
| `outDir` | 変換後の JS の出力先 |
| `rootDir` | ソースコードの場所 |
| `include` | コンパイル対象のファイル |

**重要：** `tsconfig.json` はプロジェクト全体の設定ではなく、TypeScript コンパイラの設定。`.env` や `drizzle.config.ts` はコンパイル対象に含めない。

## 12. `import` 文 vs `import()` 関数

**Q:** なぜ `import()` で `.env` を読み込めないのか？

**A:**

| 書き方 | 種類 | 動作 |
|---|---|---|
| `import Database from "..."` | import 文（宣言） | ファイル先頭で同期的に読み込む |
| `import("...")` | import() 関数（式） | 非同期で読み込む（Promise を返す） |

`.env` を読む `dotenv/config` は同期的に実行される必要があるので、import 文を使う。

---

## 13. `import` の順番が重要

**Q:** `.env` がある同じディレクトリで実行しているのに DB テーブルが見つからないのはなぜ？

**A:** `import` は書かれた順番に実行される。`dotenv/config` のインポートが、`process.env` を使うコードより**前に**来ている必要がある。

```typescript
import 'dotenv/config';           // ① .env を読み込む
import { db } from "./index";     // ② process.env.DATABASE_URL を使う
```

`db/index.ts` に `dotenv` を入れれば、`db` をインポートするすべてのファイルで自動的に `.env` が読み込まれる。

## 15. `export default` vs `export { }`

**Q:** `default` を付ける・付けないの判断は？

**A:**

| | `export default` | `export { }` (名前付き) |
|---|---|---|
| 1ファイルにつき | 1つだけ | いくつでも |
| import 時の名前 | 自由に変えられる | 元の名前を使う |
| import の書き方 | `import app from "..."` | `import { app } from "..."` |
| 用途 | ファイルのメインのもの | 複数のものを公開 |

迷ったら名前付きエクスポート（`export { }`）で統一するのが安全。

## 16. Hono のルート構成

**Q:** `index.ts` と `routes/books.ts` の関係は？

**A:**

| ファイル | 役割 |
|---|---|
| `index.ts` | アプリ全体の司令塔。サーバー起動とリクエストの振り分け |
| `routes/books.ts` | `/api/books` 以下の処理を担当するサブルーター |

`app.route("/api/books", booksRouter)` で「`/api/books` に来たリクエストはこのルーターに任せる」と登録する。Java の `@RequestMapping("/api/books")` と同じ考え方。

## 17. `app.route()` は1回だけ登録すればよい

**Q:** 各エンドポイントごとに `app.route()` を書く必要がある？

**A:** いいえ。`app.route()` は **1回だけ** 登録すれば、サブルーター内のすべてのルートがマウントされる。

```typescript
// index.ts — これだけでOK
App.route("/api/books", app);
```

これで `books.ts` の中のすべてのルートが `/api/books` の下にマウントされる：

| books.ts のルート | 実際の URL |
|---|---|
| `app.get("/")` | GET /api/books |
| `app.get("/:id")` | GET /api/books/:id |
| `app.post("/")` | POST /api/books |
| `app.put("/:id")` | PUT /api/books/:id |
| `app.delete("/:id")` | DELETE /api/books/:id |

`/add`、`/update`、`/delete` のような URL を個別に登録する必要はない。

## 18. HTTP メソッドで操作を区別する（REST API）

**Q:** 同じ URL `/:id` に GET, PUT, DELETE があったら、どれが呼ばれるかわからなくない？

**A:** HTTP リクエストには **メソッド** という情報が付いている。URL が同じでもメソッドが違えば別のルートにマッチする。

| 操作 | メソッド | URL | 意味 |
|---|---|---|---|
| 一覧取得 | **GET** | /api/books | 全件取得 |
| 詳細取得 | **GET** | /api/books/1 | 1件取得 |
| 追加 | **POST** | /api/books | 新規作成 |
| 更新 | **PUT** | /api/books/1 | 既存データを更新 |
| 削除 | **DELETE** | /api/books/1 | 削除 |

これが **REST API** の設計思想。URL はリソース（何を）、メソッドは操作（どうする）を表す。

### テスト方法

- **ブラウザの URL バー** → 常に GET リクエストになる。POST, PUT, DELETE はテストできない
- **PowerShell の `Invoke-RestMethod`** → メソッドを指定して各操作をテストできる

```powershell
# GET（取得）
Invoke-RestMethod -Method GET -Uri "http://localhost:3000/api/books"

# POST（追加）— Body にデータを付ける
Invoke-RestMethod -Method POST -Uri "http://localhost:3000/api/books" -ContentType "application/json" -Body '{"title":"本","author":"著者",...}'

# PUT（更新）
Invoke-RestMethod -Method PUT -Uri "http://localhost:3000/api/books/1" -ContentType "application/json" -Body '{"title":"更新後",...}'

# DELETE（削除）
Invoke-RestMethod -Method DELETE -Uri "http://localhost:3000/api/books/1"
```

## 19. `zValidator` のターゲット指定とミドルウェアチェーン

**Q:** `zValidator("json", schema)` で JSON ボディは検証できるが、URL のパスパラメータ（`:id`）はどう検証する？

**A:** `zValidator` の第1引数はリクエストの**どの部分**を検証するかを指定する。`"json"` 以外にも指定できる。

| 第1引数 | 検証対象 | 例 |
|---|---|---|
| `"json"` | リクエストボディ | `{ "title": "本", ... }` |
| `"param"` | パスパラメータ | `/:id` の `id` |
| `"query"` | クエリパラメータ | `?q=xxx` の `q` |

**1つのルートに複数の `zValidator` を並べられる：**

```typescript
app.put("/:id",
  zValidator("param", paramSchema),
  zValidator("json", bodySchema),
  async (c) => {
    const { id } = c.req.valid("param");
    const body = c.req.valid("json");
  }
);
```

処理の流れ：
```
リクエスト → zValidator("param") → zValidator("json") → ハンドラ
```

途中でバリデーションエラーになるとそこで止まり、400 エラーが返る。すべて通過したときだけハンドラが実行される。

このミドルウェアのチェーンは認証・認可でも同じパターンで使われる（例: `認証チェック → ロールチェック → ハンドラ`）。

## 20. 分割代入（Destructuring）の `{}` あり・なし

**Q:** `const id = obj` と `const { id } = obj` は何が違う？

**A:** `{}` をつけるとオブジェクトの**中身を取り出す**。つけないと**オブジェクトそのもの**が入る。

```typescript
const data = { id: "abc123", name: "本" };

// {} なし → オブジェクト全体
const result = data;
// result = { id: "abc123", name: "本" }

// {} あり → 中のプロパティを取り出す
const { id } = data;
// id = "abc123"

// 複数同時に取り出せる
const { id, name } = data;
// id = "abc123", name = "本"
```

`import` 文でも同じ仕組み：

| 書き方 | 意味 |
|---|---|
| `import { eq } from "drizzle-orm"` | 名前付きインポート（分割代入と同じ） |
| `import app from "./books"` | デフォルトインポート（そのまま受け取る） |

## 21. 統一レスポンス形式

**Q:** API のレスポンスはなぜ統一すべき？

**A:** エンドポイントごとにレスポンスの形がバラバラだと、クライアント（フロントエンド）側で「成功か失敗かをどう判断するか」がエンドポイントごとに変わってしまう。形式を統一すれば、常に `success` を見るだけで判断できる。

### レスポンス設計

```typescript
// 成功
{ "success": true, "data": Book | Book[] }

// エラー
{ "success": false, "error": { "message": "説明文", "code": "NOT_FOUND" } }
```

| フィールド | 役割 |
|---|---|
| `success` | 成功か失敗かのフラグ。クライアントはまずこれを見る |
| `data` | 成功時の実データ（1件 or 配列） |
| `error.message` | 人間が読むためのエラー説明 |
| `error.code` | プログラムが条件分岐に使う識別子（`"NOT_FOUND"`, `"VALIDATION_ERROR"` など） |

### 実装例

```typescript
// 成功
return c.json({ success: true, data: result });

// エラー（404）
return c.json({ success: false, error: { message: "Book not found", code: "NOT_FOUND" } }, 404);
```

第2引数の `404` は HTTP ステータスコード。ステータスコードとレスポンスボディの両方でエラー情報を伝える。

Java の Spring Boot でいう `ResponseEntity` で統一的にレスポンスを返すのと同じ考え方。

## 22. `.returning()` は常に配列を返す

**Q:** `db.insert(users).values({...}).returning()` の戻り値の型は？

**A:** `.returning()` は常に**配列**を返す。1件だけ INSERT した場合でも `[{...}]` のように配列になる。

```typescript
const user: {
    id: string;
    username: string;
    password: string;
    role: "ADMIN" | "STAFF" | "USER";
}[]  // ← [] が付いている = 配列
```

理由：Drizzle の `.values()` は1件でも複数件でも受け取れるため、`.returning()` は常に配列で統一されている。

**ホバーで型を確認する習慣が重要。** 変数にカーソルを合わせれば TypeScript が推論した型が表示される。

## 23. 分割代入でプロパティを除外する（残余パターン）

**Q:** オブジェクトから特定のプロパティだけ除きたいときは？

**A:** 分割代入と `...rest`（残余パターン / Rest Pattern）を組み合わせる。

```typescript
const user = { id: "1", username: "admin", password: "hashed...", role: "ADMIN" };

const { password, ...rest } = user;
// password → "hashed..."（取り出されるが使わない）
// rest     → { id: "1", username: "admin", role: "ADMIN" }
```

`...rest` は「残りのプロパティを全部まとめて新しいオブジェクトに入れる」という意味。レスポンスからパスワードなどの機密情報を除外するのに便利。

**重要な考え方：** `...rest` は「除外したいものを指定する」のではなく、**「取り出したいものを名前で指定し、残りを `...rest` にまとめる」** という仕組み。結果として除外に使える。

```typescript
const user = { id: "1", username: "admin", password: "xxx", role: "ADMIN" };

// 1つ取り出す → rest は残り3つ
const { password, ...rest } = user;
// rest → { id: "1", username: "admin", role: "ADMIN" }

// 2つ取り出す → rest は残り2つ
const { password, id, ...rest2 } = user;
// rest2 → { username: "admin", role: "ADMIN" }
```

`rest` は慣例的な名前で、`...other` や `...userWithoutPassword` など自由に変更可能。

## 24. bcrypt によるパスワードのハッシュ化

**Q:** なぜパスワードを平文で DB に保存してはいけない？

**A:** DB が漏洩した場合、全ユーザーのパスワードがそのまま見えてしまう。`bcrypt` でハッシュ化すれば、元のパスワードを復元できなくなる。

```typescript
import { hash, compare } from "bcryptjs";

// 保存時：ハッシュ化
const hashedPassword = await hash("admin", 10);
// → "$2a$10$..." のような文字列になる（元に戻せない）

// 照合時：compare で比較
const isMatch = await compare("admin", hashedPassword);
// → true（一致）or false（不一致）
```

| 関数 | 用途 | 戻り値 |
|---|---|---|
| `hash(平文, ソルトラウンド)` | パスワードをハッシュ化 | ハッシュ文字列 |
| `compare(平文, ハッシュ)` | パスワードが正しいか照合 | `boolean` |

第2引数の `10` はソルトラウンド（ハッシュの計算回数）。大きいほど安全だが遅い。`10` が一般的。

## 25. JWT（JSON Web Token）の仕組み

**Q:** JWT とは何？

**A:** ログイン成功の「証明書」。サーバーが発行し、クライアントが以降のリクエストに付けて送る。

```
① POST /login → サーバーが JWT を発行
② GET /books（Authorization: Bearer <JWT>）→ サーバーが JWT を検証
```

JWT は**署名されているが暗号化されていない**。中身（payload）は誰でも Base64 デコードすれば読める。だからパスワードなどの機密情報を payload に入れてはいけない。

```typescript
import { sign, verify } from "hono/jwt";

// トークン発行（login 時）
const token = await sign({ sub: user.id, role: user.role }, secret);

// トークン検証（ミドルウェアで）
const payload = await verify(token, secret, "HS256");
```

| 関数 | 用途 | 使う場所 |
|---|---|---|
| `sign(payload, secret)` | トークンを作る | login ハンドラ |
| `verify(token, secret)` | トークンを検証する | 認証ミドルウェア |

## 26. `createMiddleware` でミドルウェアを作る

**Q:** Hono でミドルウェアはどう書く？

**A:** `hono/factory` の `createMiddleware` を使う。`zValidator` と同じ位置に配置できる。

```typescript
import { createMiddleware } from "hono/factory";

export const authMiddleware = createMiddleware(async (c, next) => {
    // 前処理（認証チェックなど）
    // 失敗 → return c.json({...}, 401)
    // 成功 → await next() で次の処理へ
    await next();
});
```

ルートでの使い方：
```typescript
app.get("/", authMiddleware, async (c) => { ... });
app.post("/", authMiddleware, requireRole("ADMIN"), zValidator("json", schema), async (c) => { ... });
```

処理の流れ：`リクエスト → authMiddleware → requireRole → zValidator → ハンドラ`。途中で失敗すればそこで止まる。

## 27. 高階関数（関数を返す関数）

**Q:** `requireRole("ADMIN")` はなぜ `()` が2段階になっている？

**A:** `requireRole` は呼ぶと**ミドルウェア関数を返す**関数。引数で挙動をカスタマイズできる。

```typescript
// requireRole を呼ぶ → ミドルウェアが返る
const adminOnly = requireRole("ADMIN");
// adminOnly はミドルウェア関数

// ルートに直接書くことが多い
app.delete("/:id", authMiddleware, requireRole("ADMIN"), handler);
```

この「関数を返す関数」を**高階関数（Higher-Order Function）**と呼ぶ。ミドルウェアのカスタマイズに便利なパターン。

## 28. `c.set()` / `c.get()` によるデータの受け渡し

**Q:** ミドルウェアで検証したユーザー情報を、後続のハンドラでどう使う？

**A:** Hono の Context（`c`）に `set` / `get` でデータを保存・取得できる。

```typescript
// ミドルウェアで保存
c.set("user", payload);  // payload = { sub: "user-id", role: "ADMIN" }

// ハンドラで取得
const user = c.get("user");
console.log(user.role);  // "ADMIN"
```

同一リクエスト内でのみ有効。リクエストが終わると消える。

## 29. 401 Unauthorized vs 403 Forbidden

**Q:** どちらもアクセス拒否だが、何が違う？

**A:**

| コード | 意味 | 使う場面 |
|---|---|---|
| **401** Unauthorized | 「あなたが誰か分からない」 | トークンがない・無効・期限切れ |
| **403** Forbidden | 「あなたが誰かは分かったが、権限がない」 | ロールが足りない |

認証ミドルウェア（誰か確認）→ 失敗は **401**
認可ミドルウェア（権限確認）→ 失敗は **403**

## 30. 分割代入での変数リネーム

**Q:** `const { password, ...rest } = user` で `password` がスコープ内に既にある場合はどうする？

**A:** `:` で別名を付けられる。

```typescript
const { username, password, role } = c.req.valid("json");
// ↑ この password と被る

const { password: _unused, ...rest } = user[0];
// password の値は _unused に入り、rest に残りが入る
```

`password: _unused` は「`password` プロパティの値を `_unused` という変数名で受け取る」という意味。TypeScript の型注釈（`password: string`）とは別の構文。

## 31. NULL 許可カラムの設計

**Q:** DB カラムで「値がないかもしれない」ことを表現するには？

**A:** Drizzle では `.notNull()` を**付けない**ことで、そのカラムは NULL を許可する。

| オプション | DB の制約 | TypeScript の型 |
|---|---|---|
| `.notNull()` | NULL 不可 | `number` |
| `.notNull()` を付けない | NULL 許可 | `number \| null` |

例：貸出の `returnDate`（返却日）は、貸出中は値がなく、返却時に初めて値が入る。

```typescript
returnDate: integer("returnDate"),          // NULL 許可 → number | null
rentedDate: integer("rentedDate").notNull(), // NULL 不可 → number
```

`.default(値)` は「省略時にデフォルト値を入れる」もので、「値がない」こととは違う。

## 32. Hono の型パラメータ（Variables）

**Q:** `c.get("user")` で「型 `"user"` の引数を型 `'never'` に割り当てられない」エラーが出るのはなぜ？

**A:** Hono のルーターは、ミドルウェアが `c.set()` でどんな変数を設定するか知らない。型パラメータで教える必要がある。

```typescript
// NG: Hono は "user" というキーを知らない
const app = new Hono();

// OK: Variables で "user" の型を宣言
const app = new Hono<{ Variables: { user: { sub: string, role: string } } }>();
```

`Variables` に書いた型は、JWT の payload（`sign()` に渡したオブジェクト）と一致させる。`auth.ts` の `sign({ sub: user.id, role: user.role }, secret)` に対応して `{ sub: string, role: string }` とする。

## 33. Drizzle の JOIN（innerJoin）

**Q:** 複数テーブルのデータを結合して取得するには？

**A:** `innerJoin()` を使い、`select({})` で取得カラムを指定する。

```typescript
db.select({
    id: rentals.id,
    username: users.username,
    bookTitle: books.title,
    rentedDate: rentals.rentedDate,
}).from(rentals)
  .innerJoin(users, eq(rentals.userId, users.id))
  .innerJoin(books, eq(rentals.bookId, books.id))
  .all();
```

| メソッド | 意味 |
|---|---|
| `.innerJoin(テーブル, 条件)` | 条件に一致する行だけ結合 |
| `select({})` に書く | レスポンスに含めるカラムを選ぶ |

`select()` を引数なしで呼ぶと全カラムが返るが、`select({})` でカラムを指定すると必要なデータだけ取得でき、型も自動で推論される。

## 34. Drizzle の `and()` と `isNull()`

**Q:** 複数の WHERE 条件を AND で繋ぐには？ NULL のチェックは？

**A:** どちらも `drizzle-orm` からインポートして使う。

```typescript
import { and, isNull, eq } from "drizzle-orm";

// 3つの条件を AND で結合
db.select().from(rentals).where(
    and(
        eq(rentals.userId, userId),
        eq(rentals.bookId, bookId),
        isNull(rentals.returnDate)
    )
).get();
```

| 関数 | 意味 | SQL 相当 |
|---|---|---|
| `and(条件1, 条件2, ...)` | すべての条件を AND で結合 | `WHERE 条件1 AND 条件2` |
| `isNull(カラム)` | カラムが NULL かチェック | `WHERE カラム IS NULL` |
| `eq(カラム, 値)` | カラムが値と等しいかチェック | `WHERE カラム = 値` |

## 36. `.ts` と `.tsx` の違い

**Q:** JSX を書いたら「JSX 式は...」というエラーが出るのはなぜ？

**A:** TypeScript のファイル拡張子には2種類あり、JSX（HTML に似た構文）を使えるかどうかが異なる。

| 拡張子 | JSX | 用途 |
|---|---|---|
| `.ts` | 使えない | 型定義、ユーティリティ関数、API クライアントなど |
| `.tsx` | 使える | React コンポーネント（UI を返す関数） |

```tsx
// BookCard.tsx（.tsx なので JSX が使える）
function BookCard() {
  return <div>Hello</div>  // ← これが JSX
}

// utils.ts（.ts なので JSX は書けない）
export function formatDate(timestamp: number): string {
  return new Date(timestamp).toLocaleDateString()
}
```

**判断基準：** `return` で `<div>` や `<p>` のような HTML タグを返すファイルは `.tsx`、そうでなければ `.ts`。

## 37. React Router の基本（`BrowserRouter`, `Routes`, `Route`）

**Q:** React で URL に応じてページを切り替えるには？

| コンポーネント | 役割 | バックエンドで言うと |
|---|---|---|
| `BrowserRouter` | ルーティング機能を有効にする外枠 | `new Hono()` |

## 38. `Outlet` と `Route` のネスト（共通レイアウト）

**Q:** ログイン後のページだけにヘッダーを表示するには？

**A:** `Route` をネストし、親 Route にレイアウトコンポーネントを指定する。レイアウト内の `<Outlet />` に子ページが表示される。

```tsx
// App.tsx
<Routes>
  <Route path="/login" element={<LoginPage />} />   {/* レイアウトなし */}
  <Route element={<Layout />}>                       {/* レイアウトあり */}
    <Route path="/books" element={<BooksPage />} />
    <Route path="/rentals" element={<RentalsPage />} />
  </Route>
</Routes>
```

```tsx
// Layout.tsx
import { Outlet, Link } from 'react-router-dom'

export default function Layout() {
  return (
    <div>
      <nav>
        <Link to="/books">Books</Link>
        <Link to="/rentals">Rentals</Link>
      </nav>
      <Outlet />   {/* ← ここに子ページが入る */}
    </div>
  )
}
```

| 概念 | 意味 | バックエンドで言うと |
|---|---|---|
| `Route` のネスト | 共通レイアウトで子ページを囲む | `app.use()` でミドルウェアを適用 |
| `Outlet` | 子ページの表示位置を指定 | `await next()` で次のハンドラへ |
| `Link` | ページリロードなしの画面遷移リンク | — (フロント固有) |

親の `Route` に `path` がない（`element` だけ）場合、URLマッチングには関わらず、レイアウトとして子を囲む役割だけを持つ。

## 40. スプレッド構文 `...` でオブジェクトを展開する

**Q:** `fetch` の第2引数で `...options` と書くのは何をしている？

**A:** `options` オブジェクトの**中身をばらして**、新しいオブジェクトのプロパティとしてコピーする構文。配列の `[...arr]` と同じ `...` だが、オブジェクトに使うとキーと値が展開される。

```typescript
const options = {
  method: "POST",
  body: JSON.stringify({ username: "admin", password: "admin" }),
};

fetch(url, {
  ...options,
  headers: {
    "Content-Type": "application/json",
  },
});
```

展開後のイメージ（概念的にはこうなる）：

```typescript
fetch(url, {
  method: "POST",
  body: JSON.stringify({ username: "admin", password: "admin" }),
  headers: {
    "Content-Type": "application/json",
  },
});
```

| 書き方 | 意味 |
|---|---|
| `...options` | `options` の各プロパティ（`method`, `body`, `headers` など）をコピー |
| 後から書いた同名プロパティ | 先に展開したものを**上書き**する |

だから `apiClient` では「呼び出し側が渡した `method` や `body` を取り込みつつ、共通の `headers` を後から足す」ために使う。

```typescript
// 呼び出し側
apiClient("/api/auth/login", {
  method: "POST",
  body: JSON.stringify({ username, password }),
});

// apiClient 内部では options = { method: "POST", body: "..." }
// → ...options で method と body が fetch に渡る
```

**注意:** プロパティの**順番**が重要。`headers` を `...options` の**後**に書けば、デフォルトの `headers` で上書きできる。逆に先に `headers` を書いてから `...options` すると、呼び出し側の `headers` で共通設定が消えることがある。

## 41. JWT からフロントでユーザー情報を取り出す

**Q:** `JSON.parse(atob(token.split(".")[1]))` は何をしている？

**A:** ブラウザに保存された JWT 文字列から、**ペイロード部分だけを読んで** `sub`（ユーザーID）と `role` を取り出している。

### JWT の構造

JWT は `.` で区切られた **3つの文字列**：

```
eyJhbGci....  .  eyJzdWI....  .  SflKxwRJ....
   ヘッダー        ペイロード        署名
```

| 部分 | 中身 | フロントで使う？ |
|---|---|---|
| ヘッダー | アルゴリズムなど | 通常は不要 |
| **ペイロード** | `sub`, `role` など | **ここを読む** |
| 署名 | 改ざん検知用 | フロントでは検証しない（バックエンドが `verify`） |

バックエンドの login で入れている payload（`auth.ts`）：

```typescript
const payload = { sub: user.id, role: user.role };
```

### 1行ずつの処理

```typescript
token.split(".")[1]
```

- `split(".")` → `["ヘッダー", "ペイロード", "署名"]` の配列
- `[1]` → 真ん中のペイロード部分だけ取得

```typescript
atob(...)
```

- Base64 デコード（ブラウザ標準）
- JWT のペイロードは Base64 エンコードされた JSON 文字列

```typescript
JSON.parse(...)
```

- デコード後の文字列 `'{"sub":"...","role":"ADMIN"}'` を JavaScript のオブジェクトに変換

```typescript
setUser({ sub: payload.sub, role: payload.role })
```

- 取り出した `sub` / `role` を React の state に保存
- ヘッダーの「ログイン中のユーザー表示」やロール別 UI で使える

### なぜページ読み込み時にやるか

ログイン API のレスポンスは `{ success, data: { token } }` **だけ**。`username` は返ってこない。

トークンだけ `localStorage` に保存しているので、**リロード後**に `useEffect` でトークンを読み戻したとき、もう一度 payload から `user` を復元する必要がある。

```
① ログイン成功 → token を localStorage に保存
② ページリロード → useEffect が token を読む
③ 29-30行目 → token から user を復元 → setUser
```

### 注意

- JWT の payload は**暗号化されていない**（誰でも読める）。パスワードは入れない（学習メモ #25）
- `try/catch` で囲むのは、壊れたトークンで `JSON.parse` が失敗するため
- `login` 成功時も同じやり方で `setUser` すると一貫する（API は `data.token` のみ返す）

## 42. 認証の「復元中」フラグ（`isLoading`）でリダイレクトのちらつきを防ぐ

**Q:** ログイン済みで `/books` をリロードすると、一瞬 `/login` に飛ぶことがある。どう考えればよい？

**A:** 最初の描画では `localStorage` の token をまだ読んでいないので、`isLoggedIn` は `false` のまま。`ProtectedRoute` が即 `<Navigate to="/login" />` してしまう。**ストレージからの復元が終わるまで**リダイレクトしないフラグが必要。

| 状態 | `isLoading` | `isLoggedIn` | UI の挙動 |
|---|---|---|---|
| 復元前 | `true` | `false` | Loading（または空白）— `/login` へは飛ばさない |
| 復元後・未ログイン | `false` | `false` | `/login` へ |
| 復元後・ログイン済み | `false` | `true` | 子ルート（`<Outlet />`）を表示 |

`useEffect` で token を読み、`setIsLoggedIn` などを終えた**あと**に必ず `setIsLoading(false)` する。`if (!payload) { ...; return }` のように途中で `return` すると `setIsLoading(false)` に届かず永遠に Loading になるので、**effect の最後に1回** `setIsLoading(false)` を置くか、`if/else` で分岐してから最後に1回にまとめる。

**ログイン画面:** `isLoading` のあとに `isLoggedIn` を判定する。順序を逆にすると、復元前に一瞬ログインフォームが見える。

**用語:** ストレージやサーバーからクライアントの状態を埋めることを「ハイドレーション」と呼ぶことがある（React の SSR の「hydration」とは別文脈でも使われる）。

## 43. `useCallback` は何をしているか

**Q:** `useCallback` は何を行っている？

**A:** **関数の参照をメモ化（覚えておく）** するフック。コンポーネントが再レンダーされても、依存配列が変わらない限り **同じ関数オブジェクト** を返し続ける。

### 通常の関数定義との違い

```typescript
// 毎回の再レンダーで「新しい関数」が作られる
const fetchBooks = async (query: string) => { ... }

// 依存 [] が変わらない限り「同じ関数」を使い回す
const fetchBooks = useCallback(async (query: string) => { ... }, [])
```

React は `===` で「同じかどうか」を見る。再レンダーのたびに新しい関数ができると、`useEffect(..., [fetchBooks])` は「依存が変わった」と判断して effect を **もう一度実行** してしまう。

### 今回の BooksPage での役割

```typescript
useEffect(() => {
  void fetchBooks('').finally(() => setLoading(false))
}, [fetchBooks])
```

| `fetchBooks` の作り方 | `useEffect` の動き |
|---|---|
| 普通の `const fetchBooks = ...` | 再レンダーごとに関数が新しくなる → effect が何度も走る危険 |
| `useCallback(..., [])` | 初回と同じ関数のまま → **マウント時に1回だけ** 取得しやすい |

`fetchBooks` は引数 `query` だけ使い、`searchQuery` など外の state に依存していないので、第2引数 `[]` で「ずっと同じ関数でよい」と宣言できる。

### `useMemo` との違い（イメージ）

| フック | メモ化するもの |
|---|---|
| `useMemo` | **計算結果**（値） |
| `useCallback` | **関数そのもの** |

`useCallback(fn, deps)` は `useMemo(() => fn, deps)` とほぼ同じ意味、と覚えてもよい。

### いつ使うか

- `useEffect` / `useMemo` の依存配列に **関数** を入れたいとき
- 子コンポーネントに関数を渡し、不要な再レンダーを減らしたいとき（`React.memo` と組み合わせ）

毎回 `useCallback` が必要なわけではない。今回は **リンター警告を避けつつ、初回取得だけ走らせる** ために使った、という位置づけ。

## 44. React のフックは「条件付き return の前」に書く

**Q:** `BookEditPage` で、権限チェックの `return <Navigate />` の後に `useEffect` を書くと Biome が「This hook is being called conditionally」とエラーになるのはなぜ？

**A:** React の **Rules of Hooks（フックのルール）** に違反しているため。`useState` / `useEffect` などのフックは、**毎回のレンダーで同じ順番・同じ回数** 呼ばれないといけない。

### NG 例（今回のパターン）

```tsx
export function BookEditPage() {
  const [title, setTitle] = useState('')  // OK（毎回呼ばれる）

  if (user?.role !== 'ADMIN' && user?.role !== 'STAFF') {
    return <Navigate to="/books" replace />  // USER のときここで終了
  }

  useEffect(() => { ... }, [id])  // ADMIN/STAFF のときだけ呼ばれる → 違反
}
```

| ロール | `useEffect` が実行される？ |
|---|---|
| USER | 呼ばれない（手前で return） |
| ADMIN / STAFF | 呼ばれる |

→ レンダーごとにフックの数が変わるのでエラーになる。

### OK 例

```tsx
export function BookEditPage() {
  const { user } = useAuth()
  const { id } = useParams()
  const [title, setTitle] = useState('')
  // ... 他の useState

  useEffect(() => {
    if (!id) return
    void (async () => { /* GET */ })()
  }, [id])

  // フックをすべて書いた「後」に条件付き return
  if (user?.role !== 'ADMIN' && user?.role !== 'STAFF') {
    return <Navigate to="/books" replace />
  }

  return ( /* フォーム */ )
}
```

### 覚え方

```
useState / useEffect / useContext ...  ← 先にまとめて書く
↓
if (...) return <Navigate />          ← 権限チェックなど
↓
if (loading) return ...
return ( JSX )
```

- **フックの前に `return` してはいけない**（その `return` によってフックがスキップされる場合）
- `useEffect` の**中の** `if (!id) return` は OK（フック自体は毎回呼ばれている）

### 補足（BookEditPage でよくある動作バグ）

フックのエラーとは別に、編集画面では次も注意する。

1. **GET 成功後** — `setBook` だけでなく `setTitle(book.title)` など、フォーム用 state も更新する
2. **`loading`** — 取得完了前に `book === null` で「Book not found」と出ないようにする

## 45. 「返却ボタン」が表示される画面の設計

**Q:** 返却ボタンが出ないのはなぜ？

**A:** 現在の設計では、返却ボタンは **管理用一覧（`/rentals`）** にだけ表示し、**自分の貸出（`/rentals/my`）** には表示しない作りになっているため。

| 画面 | 対象ロール | 返却ボタン |
|---|---|---|
| `/rentals` | ADMIN / STAFF | 表示する（未返却の行だけ） |
| `/rentals/my` | 全ロール | 既定では表示しない |

設計意図は「管理者・スタッフが全体の返却処理を行う」か「各ユーザーが自分で返却できる」かで変わる。どちらでも API は同じ `PUT /api/rentals/:id/return` を使えるので、要件に合わせて UI だけ出し分ける。

**確認ポイント:**

1. 今見ているページが `/rentals/my` なら、返却ボタンがなくても仕様どおりの可能性が高い
2. `/rentals` を ADMIN / STAFF で開き、`returnDate === null` の行にボタンが出るか確認する
3. 全ユーザーに返却を許可したい場合は `/rentals/my` にも同じ条件でボタンと `handleReturn` を追加する
